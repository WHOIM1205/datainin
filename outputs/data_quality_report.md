# Key data-integrity findings (documentation only)

1. **Numeric `Timestamp` column is LOSSY** — only 7 unique values across 211,224 rows (rounded to ~3 significant figures). It is unusable for time. **`Timestamp IST` (string, IST tz) is the authoritative time field.**
2. **`Trade ID` is NOT a unique key** — only 2,810 unique values across 211,224 rows, and values appear rounded (e.g. 895000000000000). Full-row duplicates = 0, so rows are distinct; but no single reliable trade-level key exists (Order ID and Transaction Hash are one-to-many across fills).
3. **No `leverage` column** — the PDF listed it, but the file has none. Leverage must be either derived as a documented proxy (e.g. from Size USD vs Start Position) or excluded, with the decision justified in the cleaning module.
4. **`Closed PnL` is sparse** — 50.6% of rows have PnL = 0 (realized PnL booked on closing fills only). Win-rate must therefore be defined on PnL-bearing (closing) trades, not all fills — to be decided in the metrics module.
5. **`Direction` has 12 values** including edge cases (Auto-Deleveraging, Liquidated, Settlement, Long > Short, Spot Dust Conversion). Long/short classification logic must handle these explicitly.
6. **Boundary values present** — negative `Fee` rows: 2,476 (maker rebates); zero `Execution Price` rows: 0; zero `Size USD` rows: 43. Flagged for review in cleaning; not touched here.
7. **Account concentration** — only 32 unique accounts and 246 unique coins; HYPE alone is 32.2% of rows. Market-level aggregates will be dominated by a few accounts/coins — relevant for interpretation.
